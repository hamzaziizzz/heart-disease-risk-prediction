# Heart Disease Risk Prediction — End-to-End MLOps Pipeline

**Course:** Machine Learning Operations (MLOps) — AIMLCZG523
**Programme:** BITS Pilani WILP M.Tech. Artificial Intelligence & Machine Learning
**Assignment:** Assignment 01 — MLOps Experimental Learning Assignment (50 marks)

An end-to-end, production-style ML system that predicts heart disease risk from patient
clinical data: data pipeline → EDA → feature engineering → model training with MLflow
tracking → packaging → automated testing → CI/CD → Docker containerization → Kubernetes
deployment → Prometheus/Grafana monitoring.

## 2. Quickstart — Local Setup (run from the repo root)

```bash
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# 1. Download + clean the dataset
python src/download_data.py

# 2. Train models, run GridSearchCV + 5-fold CV, log to MLflow, save best model
python src/train.py

# 3. Run the test suite
python -m pytest tests/ -v

# 4. Lint
python -m flake8 --max-line-length=110 --extend-ignore=E203,W503 src/ api/ tests/

# 5. Launch the API locally (without Docker)
uvicorn api.main:app --reload --port 8000
# Swagger UI: http://localhost:8000/docs

# 6. Browse experiment tracking results
mlflow ui --backend-store-uri sqlite:///mlflow.db
# MLflow UI: http://127.0.0.1:5000
```

## 4. Kubernetes (Docker Desktop Kubernetes)

Enable Kubernetes in Docker Desktop first (Settings → Kubernetes → Enable Kubernetes).

```bash
# Docker Desktop K8s shares the local Docker image cache, so no registry push needed --
# just make sure the image was built locally (step 3 above) before applying manifests.

kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

kubectl get pods -l app=heart-disease-api
kubectl get svc heart-disease-api-service

# The LoadBalancer Service is reachable at localhost:8082 on Docker Desktop
# (mapped to 8082 instead of the more common 8080 only because 8080 was
# already taken by another local process on the reference dev machine --
# change k8s/service.yaml's `port:` back to 8080 if that's free on yours)
curl http://localhost:8082/health
```

## 5. Monitoring (Prometheus + Grafana)

```bash
cd monitoring
docker compose -f docker-compose.monitoring.yml up -d
```

- Prometheus: http://localhost:9090 → **Status → Targets** should show `heart-disease-api` as `UP`
- Grafana: http://localhost:3000 (login `admin` / `admin`) → add a Prometheus data source
  at `http://prometheus:9090` → import `monitoring/grafana-dashboard.json`

> By default `monitoring/prometheus.yml` scrapes `host.docker.internal:8000` (the API
> running via `docker run -p 8000:8000`). If you're monitoring the Kubernetes deployment
> instead, change the target to `host.docker.internal:8082`.

## 6. CI/CD

`.github/workflows/ci.yml` runs automatically on every push/PR to `main`:
lint (flake8) → download data → train models → run pytest → upload model/plot artifacts
→ Docker build validation. The pipeline fails the build (red X on GitHub) if lint, tests,
or the Docker build fail.

## 8. Author's Note on Personalization

Per the assignment's academic-integrity requirements, this implementation makes several
deliberate, individual design choices worth calling out in case of similarity review:
median (not mean) imputation for numeric clinical features to reduce outlier sensitivity,
a 3-model comparison (Logistic Regression + Random Forest + XGBoost, rather than the
minimum 2), SQLite (not the deprecated plain-file store) as the MLflow backend,
`prometheus-fastapi-instrumentator` for zero-boilerplate Prometheus integration, and a
Docker-Desktop-Kubernetes-first deployment target (rather than a public cloud) chosen
deliberately for reproducibility without cloud credentials.
